## 2. TTS朗读失败提示分离

**位置**：AI对话/朗读功能

**当前状态**：两种失败都用相同的提示

**位置**：AI对话总结功能

- 不再放到目录上面

**当前状态**：总结内容存放在目录中

---

## 4. 会话数量上限

**位置**：会话管理

**需求**：
- 会话数量设置上限
- 上限可在配置文件中配置
- 默认上限：**10个**
- 达到上限后不能再创建新会话
- 必须先删除现有会话才能创建新的

**配置项建议**：
```yaml
session:
  max_count: 10  # 会话数量上限，默认10
```

**当前状态**：无限制

---

## 5. 总结功能使用自定义 AI 后端

**位置**：TTS 总结功能 (`internal/speech/minimax.go`)

**现状**：
- 总结功能使用 `mmx` CLI 工具调用 MiniMax
- 命令：`mmx text chat --system <prompt> --messages-file - --model <model>`

**需求**：
- 支持使用项目自己的 AI 后端进行总结
- 在配置文件中可配置使用的后端

**配置项建议**：
```yaml
tts:
  # 总结功能使用的后端类型: "minimax" (默认, 使用mmx CLI) 或 "cli" (使用项目AI后端)
  summarize_backend: "minimax"
  # 当 summarize_backend 为 "cli" 时使用的后端名称 (如 "assistant", "coder" 等)
  summarize_backend_name: "assistant"
```

**涉及文件**：
- `internal/speech/minimax.go` — MiniMaxProvider 结构体和 Summarize 方法

## 9. AI 消息自定义格式展示

**位置**：AI 对话消息渲染

**需求**：根据不同类型的消息内容，使用适合的格式展示：

| 消息类型 | 展示格式 |
|---------|---------|
| Edit / 文件编辑 | Git diff 比较报告样式（点击可跳转至对应文件） |
| Bash / 命令执行 | 终端样式（带 `$` 前缀、输出高亮） |
| Read / 文件读取 | 代码块样式（支持全文编辑） |
| Write / 文件写入 | 代码块样式（支持全文编辑） |
| Grep / 搜索 | 结果列表样式 |
| 其他 | 默认样式 |

**当前状态**：所有消息统一使用相同的格式展示

## 18. TTS 失败后重试跳过总结

**位置**：朗读功能

**需求**：
- TTS 转换失败后，缓存已生成的总结内容

- 避免重复调用总结 API，节省时间

**当前状态**：每次点击都会重新走总结流程
